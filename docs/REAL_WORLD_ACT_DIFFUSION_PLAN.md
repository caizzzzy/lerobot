# 真实环境 ACT / Diffusion 数据采集、训练与真机部署计划

## 1. 目标

这份计划面向你当前的仓库现状：

- 你已经有另外一个项目可以做遥操作采集。
- 你当前会用 `convert_to_lerobot*.py` 把原始数据转成 LeRobotDataset。
- 你已经能训练 ACT 和 Diffusion。
- 你当前的难点不是“能不能跑通”，而是“怎么把数据、训练和部署链路整理成更稳定、更有效的闭环”。

本文目标是给出一份可执行计划，帮助你在真实环境下逐步训练出效果更好的 ACT / Diffusion policy，并降低真机部署踩坑概率。

## 2. 先定三件事

在真正开始大规模采集前，先固定下面三件事。不要边采边改。

### 2.1 固定任务定义

每次只做一个明确任务，不要一开始就追求“大而全”。

任务描述要满足：

- 起始状态范围可控
- 成功标准明确
- 结束条件明确
- 失败类型可枚举

例子：

- “把桌面上的单个方块抓起并放到固定盒子中”
- “从固定托盘中抓取苹果并放到指定区域”

不建议一开始就采：

- 多目标混合任务
- 起始状态变化极大、成功定义模糊的任务
- 需要复杂语义理解的任务

### 2.2 固定动作空间

你当前仓库里同时存在几种动作/状态表达：

- 关节空间 `joint + gripper`
- TCP quaternion
- TCP 6D
- sensor gripper control / gripper observation 分离

建议先只选一种，优先级建议如下：

1. 如果你当前 Diana 真机控制最稳定的是关节控制，就先统一成 `joint + gripper`。
2. 如果你明确要做末端位姿控制，再单独开一条 `tcpquat` 或 `tcp6d` 线，不要和 joint 线混采。

原则：

- 一条训练线只对应一种 `observation.state` / `action` 定义
- 不同动作空间不要混到同一个数据集
- 训练配置、转换脚本、推理脚本三者必须严格一致

### 2.3 固定观测配置

你当前最常见的是：

- `cam_high`
- `cam_global`
- `observation.state` 8 维

建议先固定：

- 相机数量
- 相机名字
- 相机分辨率
- 采样频率
- state/action 维度

不要在同一数据集里混入：

- 一部分 episode 是两路相机，一部分是一路
- 一部分是 320x240，一部分是 640x480
- 一部分是 10Hz，一部分是 30Hz

这些变化会直接增加训练和部署不一致风险。

## 3. 该采什么数据

## 3.1 以“成功示范”为主，但不要只有完美轨迹

对于 ACT 和 Diffusion，这一阶段的主数据仍然应该是专家遥操作示范。

建议组成：

- `70%~85%` 干净成功轨迹
- `15%~30%` 带轻微修正、恢复、重试的成功轨迹

建议保留的“非完美但有价值”的数据：

- 抓取偏了一点，但通过二次修正成功
- 接近目标后姿态微调
- 放置前短暂停顿再修正

不建议大量保留的内容：

- 完整失败且没有恢复的长轨迹
- 操作者明显分神、动作乱晃的数据
- 卡顿、延迟、机械臂异常时的数据

原因：

- 纯“完美演示”容易导致模型不会恢复
- 纯“混乱失败数据”会污染动作分布

## 3.2 覆盖真实部署时会遇到的变化

每个任务都要主动覆盖以下变化，而不是只采“最理想场景”。

### 场景变化

- 物体位置有小范围随机
- 物体朝向有小范围随机
- 放置目标区域有小范围随机
- 桌面背景、光照、阴影有轻微变化

### 视觉变化

- 白天 / 夜晚 / 室内灯不同亮度
- 轻微遮挡
- 夹爪接近物体时局部视角变化

### 动作变化

- 接近路径不完全一样
- 抓取点略有不同
- 放置姿态有小变化

### 初始状态变化

- 机械臂起始姿态不要永远完全一致
- 但变化要在安全工作空间内

原则：

- 要有变化，但不是完全随机
- 变化范围要接近你未来真机部署的实际分布

## 3.3 需要多少数据

这是经验性建议，不是硬阈值。

### 单阶段、单物体、相对简单任务

起步建议：

- `50~100` 条高质量成功轨迹可以先做首轮 baseline

想得到更稳定的结果：

- `150~300` 条更稳

### 多阶段任务

例如：

- 接近 -> 抓取 -> 抬起 -> 搬运 -> 放置

建议：

- `200~500+` 条成功轨迹更现实

### ACT 与 Diffusion 的数据量倾向

- ACT：
  - 可以先用较少数据出 baseline
  - 对高质量、动作一致性要求高
  - 长时序 chunk 学得好不好，和演示风格一致性关系很大
- Diffusion：
  - 往往更吃数据量和覆盖范围
  - 对恢复动作、局部修正、观测变化通常更友好
  - 如果数据太少且变化太大，容易学得不稳

实操建议：

- 第 1 阶段：先用 `50~100` 条做 ACT 和 Diffusion 的小基线
- 第 2 阶段：根据真机失败模式补采到 `150~300`
- 第 3 阶段：只围绕失败区域定向补数据，而不是无限盲采

## 4. 采集时必须记录好的内容

## 4.1 必须稳定的字段

你的转换脚本最终至少要稳定产出：

- `observation.state`
- `action`
- `observation.images.cam_high`
- `observation.images.cam_global`
- `timestamp`
- `episode_index`
- `frame_index`

重点检查：

- 每条 episode 维度一致
- `state` 和 `action` 维度永远一致
- 相机键名永远一致
- 分辨率永远一致
- FPS 永远一致

## 4.2 强烈建议额外保存的元信息

如果你的原始采集项目里方便加，建议额外保存：

- 任务名
- 采集日期
- 操作者 ID
- 物体类别
- 场景编号
- 是否成功
- 失败类型
- 备注

哪怕这些字段暂时不进入训练，也对后续筛选数据非常有用。

## 4.3 采集频率建议

你当前仓库里已经有 10Hz 和 30Hz 两条思路。

建议：

- 如果你的真机控制和推理预算比较紧，先优先做 `10Hz` 稳定线
- 如果动作需要更细、更快的修正，再考虑 `20~30Hz`

原则：

- 采集频率尽量接近部署频率
- 不要采 30Hz 再硬降成 10Hz，除非你明确验证过这样更稳
- 不要训练时 10Hz，部署时强行按 30Hz 跑

## 5. 数据采集注意事项

## 5.1 操作者风格要尽量一致

最常见的问题之一是：数据是“人能做出来”，但动作风格太散，模型学不稳。

建议：

- 一个任务尽量由少数固定操作者采
- 先统一操作规范
- 尽量避免有人动作特别快、有人特别慢、有人喜欢抖动修正

尤其对 ACT：

- 风格一致性通常很重要

## 5.2 采“恢复动作”，但不要采“混乱动作”

应该采：

- 接近失败后稳妥修回来的动作

不该采：

- 因延迟、碰撞、机械异常导致的乱动作
- 明显失控还继续硬拖几十秒

## 5.3 起始状态要有分布，但要可控

建议每次 episode 前随机化：

- 物体位置
- 物体朝向
- 机械臂起始位姿

但变化范围要受控：

- 保证仍在相机视野内
- 保证仍在安全工作空间内
- 保证仍是人能稳定完成的任务

## 5.4 失败数据不要整包混入

建议把 episode 先分三类：

- A 类：干净成功
- B 类：有修正但成功
- C 类：失败

训练第一轮建议：

- 先只用 A + B

后面若要用 C：

- 只引入“高价值失败片段”
- 不要整包混入所有失败 episode

## 5.5 相机和标定问题要优先解决

部署效果一般时，很多时候根因不是模型，而是观测不一致。

重点检查：

- 相机视角是否稳定
- 相机编号是否会漂移
- 训练和部署时相机名字是否一致
- 训练和部署时图像尺寸是否一致
- 机械臂 / gripper 状态是否和图像同步

如果这里不稳定，后面训练再多也很难解决。

## 6. 采完之后的质检流程

每批数据采完后都做一次质检，不要等训练失败再回头看。

建议流程：

1. 随机抽 10~20 条 episode，看视频是否正常。
2. 检查 parquet 列名、shape、fps 是否一致。
3. 检查 `observation.state` 和 `action` 是否存在跳变异常。
4. 检查成功率统计。
5. 按失败类型做简单标签。

建议重点看：

- [check_data_accuracy.py](../check_data_accuracy.py)
- [look_parquet.py](../look_parquet.py)
- [test_video.py](../test_video.py)
- [src/lerobot/scripts/lerobot_dataset_viz.py](../src/lerobot/scripts/lerobot_dataset_viz.py)

## 7. 训练策略建议

## 7.1 不要一开始就同时大改模型和数据

建议顺序：

1. 先固定一版数据定义
2. 跑 ACT baseline
3. 跑 Diffusion baseline
4. 真机试
5. 根据失败模式补数据
6. 再调超参

不要这样做：

- 一边改转换脚本
- 一边改 state 定义
- 一边改相机
- 一边改 policy 超参

这样最后很难知道到底哪一步起作用。

## 7.2 ACT 和 Diffusion 的推荐试验顺序

### 第一轮

- 用最干净的数据集先训 ACT
- 同一份数据再训 Diffusion

目的：

- 建立两个最小可用 baseline

### 第二轮

根据真机失败模式决定补哪类数据：

- 如果模型不够稳、容易偏离但还能修：
  - 补“恢复成功”的轨迹
- 如果模型对位置变化敏感：
  - 补初始状态随机化
- 如果模型对光照/视角敏感：
  - 补视觉分布变化

### 第三轮

再考虑是否：

- 提高分辨率
- 调整 FPS
- 改 action 表示
- 换到 TCP 表示

## 7.3 数据切分建议

不要随机按 frame 切分，建议按 episode 切分。

推荐：

- train: `80%`
- val: `10%`
- test / holdout: `10%`

更好的做法：

- 按采集日期或场景切一个真正 holdout 集
- 用它专门判断泛化，不参与调参

## 8. 当前仓库下的推荐训练线

如果你要先走最稳的一条线，建议如下。

### 8.1 第一条主线

- 动作空间：`joint + gripper`
- 相机：`cam_high + cam_global`
- 分辨率：统一 `320x240` 或统一 `640x480`，不要混
- 频率：优先 `10Hz` 稳定线
- 任务：单任务、单物体、单工作空间

原因：

- 这条线和你当前的 Diffusion 推理脚本最接近
- 部署复杂度相对最低
- 最容易先形成闭环

### 8.2 ACT 训练建议

适合先拿来做：

- 较短时长
- 操作风格比较一致
- 演示较规范

注意：

- `chunk_size` 不宜一开始设得太大
- 如果动作风格差异很大，ACT 可能学得不稳定

### 8.3 Diffusion 训练建议

适合先拿来做：

- 有一定观测变化
- 需要一定恢复能力
- 多步动作耦合较强

注意：

- `n_obs_steps`、`n_action_steps`、部署频率必须一致思考
- 数据量太少时，Diffusion 往往不如你预期稳定

## 9. 真机部署前检查清单

在把模型真正下到机器人之前，先逐项确认：

- 训练时和部署时的 `observation.state` 定义完全一致
- 训练时和部署时的 camera key 完全一致
- 分辨率一致
- FPS 一致
- 归一化统计来自正确数据集
- checkpoint 对应的数据集版本没有搞错
- 相机画面和 state 同步正常
- 机器人安全限位、速度限制、急停可用

额外建议：

- 先做空场景 dry-run
- 先让机器人只执行少量步数
- 先低速
- 先限定工作空间

## 10. 真机部署迭代闭环

推荐你用下面这个闭环，而不是“采很多 -> 训很久 -> 一次性上机”。

### 第 0 轮

- 采 `50~100` 条高质量成功数据
- 训练 ACT + Diffusion baseline
- 做小规模真机测试

### 第 1 轮

记录失败类型，至少分成：

- 没看见目标
- 接近路径不对
- 抓取姿态不对
- 闭合时机不对
- 搬运中掉落
- 放置偏移
- 恢复能力差

### 第 2 轮

针对失败类型定向补采：

- 每类失败补 `20~50` 条有针对性的成功/恢复数据

### 第 3 轮

重新训练，观察：

- 成功率是否提升
- 失败类型是否收敛
- 是否出现新失败模式

### 第 4 轮

只有在数据已经比较稳定后，再考虑：

- 调模型
- 改动作空间
- 加更多相机
- 升分辨率
- 提高 FPS

## 11. 一个可执行的近三周计划

## 第 1 周：固定协议 + 小样本 baseline

- 明确任务定义和成功标准
- 固定动作空间、相机键名、分辨率、频率
- 采 `50~100` 条高质量成功轨迹
- 用转换脚本转成 LeRobotDataset
- 做一次 parquet/video 质检
- 跑 ACT baseline
- 跑 Diffusion baseline

输出：

- 一版可复现数据集
- 一版 ACT checkpoint
- 一版 Diffusion checkpoint
- 一份真机失败记录

## 第 2 周：围绕失败补数据

- 按失败类型分类
- 每类失败补 `20~50` 条针对性数据
- 增加轻微场景随机化
- 清理明显低质量 episode
- 重新训练

输出：

- 第二版数据集
- 第二版 checkpoint
- 失败类型对比表

## 第 3 周：稳定部署线

- 固定一条最稳部署链路
- 做小规模连续真机评估
- 对比 ACT / Diffusion 在同一任务上的差异
- 决定后续主力模型

输出：

- 当前主力训练配置
- 当前主力转换脚本
- 当前主力部署脚本

## 12. 当前阶段最重要的结论

如果只保留几条最重要建议，就是这几条：

1. 先固定一条单任务、单动作空间、单双相机稳定线，不要混采。
2. 先做 `50~100` 条高质量 baseline 数据，不要一开始盲目追求海量。
3. 先用 A/B 类成功轨迹训练，不要把大量失败轨迹直接灌进去。
4. 真机效果一般时，优先怀疑数据定义、观测一致性、部署一致性，而不是先怀疑模型不够大。
5. 用“小批量采集 -> 训练 -> 真机试 -> 按失败补数据”的闭环迭代，通常比一次性堆数据更有效。
